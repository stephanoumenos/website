When I first learned Go, interfaces felt limiting. No generics (at the time), no inheritance,
no abstract classes. Just interfaces with a handful of methods.

Then I understood implicit satisfaction, and everything clicked.

## Implicit, not explicit

In Java or C#, you declare that a type implements an interface:

```java
class MyWriter implements Writer { ... }
```

In Go, you don't. A type satisfies an interface simply by having the right methods:

```go
type Writer interface {
    Write(p []byte) (n int, err error)
}

// MyWriter satisfies Writer — no declaration needed.
type MyWriter struct{}

func (w MyWriter) Write(p []byte) (n int, err error) {
    // ...
    return len(p), nil
}
```

This one decision has large consequences for how you design APIs.

## What it unlocks

Because Go interfaces are implicit, you can define an interface *at the point of use* rather than
at the point of implementation. This is the crucial thing.

Say you're writing a function that needs to send a notification. Instead of importing some
`Notifier` interface defined by a dependency, you define exactly what you need:

```go
type Sender interface {
    Send(ctx context.Context, msg string) error
}

func notify(s Sender, msg string) error {
    return s.Send(context.Background(), msg)
}
```

Any type with a `Send` method works — whether it's your email client, your Slack webhook, or a
test fake. The function owns its dependency contract.

## The design principle

This leads to a principle I now follow consistently: **keep interfaces small and define them
where they're used, not where they're implemented.**

The standard library models this perfectly. `io.Reader` has one method. `io.Writer` has one method.
`io.Closer` has one method. They compose:

```go
type ReadWriteCloser interface {
    Reader
    Writer
    Closer
}
```

When you define a function that takes an `io.Reader`, it works with files, network connections,
bytes buffers, test readers — anything. The interface is so minimal that almost anything satisfies it.

## Testing becomes effortless

The biggest practical benefit: testing. When your functions depend on small interfaces rather than
concrete types, you can write test fakes in three lines:

```go
type fakeSender struct{ sent []string }

func (f *fakeSender) Send(_ context.Context, msg string) error {
    f.sent = append(f.sent, msg)
    return nil
}
```

No mocking library needed. No reflection. Just a struct that satisfies the interface your function
needs.

---

Go's interfaces taught me to think about dependencies differently. The question isn't "what does
this type provide?" but "what does this function need?" It's a subtle shift, but once you feel it,
you can't go back.
